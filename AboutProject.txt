1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Orbit AI is a lightweight, AI-powered desktop code editor / IDE built on
Electron. It provides a modern development environment with:

- A full-featured code editor powered by Monaco (the engine behind VS Code)
- An integrated terminal (PowerShell on Windows)
- A powerful AI coding assistant sidebar powered by Groq LLMs

The application enables developers to open any local folder, browse files,
edit code with syntax highlighting and IntelliSense, execute shell commands,
and receive contextual AI assistance — all within a single desktop window.

Target Users:
- Individual developers and hobbyists who want AI assistance without using
  heavy IDEs or cloud-based editors.
- Users who prefer a fast, native desktop experience with full local file
  system access.
- Developers working on Windows who want a simple yet capable AI-augmented
  editor.

Why It Exists:
Orbit AI was created to explore building a practical, privacy-friendly,
fully-local AI coding companion that combines the power of modern LLMs
(Groq) with familiar desktop editing workflows, without requiring users to
upload entire projects to third-party platforms.

4. BACKEND ARCHITECTURE
--------------------------------------------------------------------------------

There is no traditional backend server. All server-side logic runs inside
Electron's Main Process (Node.js environment).

Main Process Entry:
- src/index.ts — Contains all main-process logic

Responsibilities of the Main Process:
1. Application lifecycle (app.on('ready'), window-all-closed, activate)
2. BrowserWindow creation with secure webPreferences:
   - contextIsolation: true
   - nodeIntegration: false
3. IPC handlers registered via ipcMain.handle() / ipcMain.on()
4. Native OS dialogs (folder picker)
5. File system operations using Node fs/promises
6. Recursive directory tree generation with sensible ignores
7. Child process spawning for integrated terminal (PowerShell via node-pty)
   - Special handling required: node-pty marked as webpack external + lazy runtime require()
   - This pattern was necessary due to conpty.node loading issues in bundled main process on Windows
   - Known limitation: PowerShell + conpty can exit immediately in very large folders (root projects, folders with heavy node_modules). Likely caused by antivirus interference or Windows path/conpty initialization limits. Subfolders usually work reliably. Improved error reporting added to surface real exit codes.
8. Direct calls to Groq AI via the official SDK (no proxy)

IPC Communication Pattern:
Renderer (preload)  <—(invoke/send)→  Main Process (ipcMain)
Main Process        —(send)→          Renderer (via webContents)

All privileged operations (file access, spawning processes, calling AI APIs)
are performed exclusively in the main process and exposed to the renderer
through a narrow, audited API surface in preload.ts.

Security Model:
- Renderer has zero direct Node.js or filesystem access.
- All capabilities are explicitly bridged via contextBridge.
- Fuses are configured in forge.config.js to further harden the packaged app.

Data Flow Examples:
- User clicks "Open Folder" → dialog:openFolder → native dialog → returns path
- User clicks file in tree → fs:readFile → returns content → Monaco editor
- User sends chat message → ai:stream → Groq streaming → chunks pushed via
  'ai:stream-chunk' events → AIChat component renders tokens live
- User types in terminal → terminal:write → stdin of PowerShell child process

The architecture follows Electron security best practices and keeps the
attack surface small by using a strict preload bridge.


6. DATABASE
--------------------------------------------------------------------------------

Orbit AI does not use any database.

All persistent data lives on the user's local filesystem:
- Source code files being edited
- The opened project folder itself

The only piece of application state that survives restarts is the Groq API
key, which is stored in the browser's localStorage under the key
'groq_api_key'.

There are no SQLite, IndexedDB (beyond localStorage), or any other data
storage layers.

8. API DOCUMENTATION
--------------------------------------------------------------------------------

Orbit AI exposes no public HTTP or REST API.

All inter-process communication happens through a tightly-scoped Electron
contextBridge API defined in src/preload.ts and exposed as
window.electronAPI.

Exposed API Surface (window.electronAPI):

File System & Dialogs
- openFolder(): Promise<string | null>
- readFile(path: string): Promise<string>
- writeFile(path: string, content: string): Promise<boolean>
- createFile(path: string, content?: string): Promise<boolean>
- createDirectory(path: string): Promise<boolean>
- getFileTree(folderPath: string): Promise<FileNode[]>

Terminal
- terminalCreate(cwd?: string): Promise<{ pid: number }>
- terminalWrite(data: string): void
- terminalResize(cols: number, rows: number): void
- onTerminalData(callback: (data: string) => void): void

AI (Groq)
- aiChat(apiKey: string, messages: Message[]): Promise<string>
- aiStream(apiKey: string, messages: Message[]): Promise<string>
- onStreamChunk(callback: (delta: string) => void): void

FileNode interface:
{
  name: string;
  type: 'file' | 'directory';
  path: string;
  children?: FileNode[];
}

AI Message format:
{ role: 'system' | 'user' | 'assistant'; content: string }

Note: Streaming is the primary AI interaction path used by the UI.
The non-streaming aiChat handler exists but is not used in the current
frontend implementation.


9. AUTHENTICATION & AUTHORIZATION
--------------------------------------------------------------------------------

Application-Level Authentication:
- None. The desktop application does not have users, accounts, or login.

Authorization / Permissions:
- The app runs with whatever permissions the user who launched it has on
  their local machine.
- Full read/write access to any folder the user explicitly opens via the
  native dialog.

Groq AI Authentication:
- Users must provide their own Groq API key via the UI (🔑 button in chat).
- The key is stored only in localStorage (never sent anywhere except directly
  to Groq's API from the main process).
- No key is bundled with the application.
- Link to https://console.groq.com/keys is provided in the interface.

There is no role-based access control, session management, or token refresh
mechanism because none of these concepts apply to a single-user desktop tool.

12. EXTERNAL INTEGRATIONS
--------------------------------------------------------------------------------

1. Groq AI (Primary AI Provider)
   - Endpoint: https://api.groq.com (via groq-sdk)
   - Models used:
     • llama3-8b-8192 (non-streaming handler)
     • llama-3.3-70b-versatile (streaming, primary chat model)
   - Authentication: User-supplied API key sent directly from main process
   - Data sent: Current file path + truncated file content (max ~8000 chars)
                 as system prompt, plus conversation history
   - Purpose: Real-time code assistance, explanations, and code generation

2. Local Filesystem
   - Full read/write/create access to any directory the user opens
   - Recursive directory walking (depth-limited to 8 levels for safety)
   - Used for: project browsing, editing, and saving files

3. PowerShell (Windows Shell)
   - Executable: powershell.exe
   - Spawned via Node child_process.spawn
   - Working directory defaults to user profile or the opened folder
   - Full environment variables are inherited
   - Used exclusively for the integrated terminal panel
   - Note: Currently Windows-only (no macOS/Linux shell fallback)

4. Monaco Editor Workers
   - Loaded dynamically from monaco-editor package
   - Provides language services for TypeScript, JavaScript, JSON, etc.

No other third-party services, analytics, crash reporting, or update servers
are integrated at this time.
